1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the NoWait Smart Hospital System developed by Auralith Bit. The document serves as the authoritative reference for developers, designers, testers, project managers, and hospital stakeholders.
1.2 Scope
NoWait is a lightweight, web/mobile-based Smart OPD (Outpatient Department) Booking and Queue Management System designed specifically for hospitals, clinics, and health centers in Nepal. The system aims to:
Eliminate 2–5 hour physical queues in hospital OPDs
Digitize appointment booking via web, mobile, and WhatsApp
Provide real-time token/queue tracking to patients
Improve hospital staff workflow and doctor scheduling
Deliver analytics to hospital administrators

2. Overall System Description
2.1 Product Perspective
NoWait is a standalone, cloud-hosted SaaS platform. It can optionally integrate with existing Hospital Management Information Systems (HMIS) via REST API. Initially, it operates independently using a browser/mobile interface and requires no existing IT infrastructure at the hospital.
2.2 Problem Statement
PROBLEM: Nepali hospitals face severe OPD congestion. Patients wait 2–5 hours in physical queues.
IMPACT:  Patient dissatisfaction, staff overwhelm, revenue loss from no-shows.
SOLUTION: A digital queue + booking system that gives patients a virtual token and notifies them when their turn is near.

2.4 Operating Environment
Frontend: Progressive Web App (PWA) — runs on Android, iOS, and Desktop browsers
Backend: Cloud-hosted (AWS / GCP / DigitalOcean) — Node.js or Python
Database: PostgreSQL (primary) + Redis (queue/real-time state)
SMS Gateway: Sparrow SMS (Nepal) or InfoBip
Internet: 2G/3G/4G compatible; designed for low-bandwidth environments
Languages: Nepali + English (bilingual interface)
2.5 Design & Implementation Constraints
Must work offline partially (token info accessible without internet after booking)
No native app required at MVP — PWA only
Must support WhatsApp notification (phase 2) for non-smartphone users
Low-cost initial infrastructure — can start with Google Sheets + simple backend
Patient data privacy as per Nepal's Electronic Transaction Act (ETA) 2063
3. Functional Requirements
3.1 Patient Module
3.1.1 Registration & Login
Patient can register using mobile number + OTP
Optional: name, date of birth, gender, blood group
Login via OTP (no password required — phone-number first)
Guest booking allowed without registration (phone number required)
3.1.2 OPD Appointment Booking
Patient selects: Hospital → Department → Doctor → Date → Time Slot
System shows available slots in real time
Patient receives a unique Token Number upon booking
Confirmation sent via SMS with token, doctor name, time, and location
Patient can book on behalf of family member (dependent profile)
Booking cancellation allowed up to 1 hour before slot
3.1.3 Walk-in Token (Physical Queue)
Patients who arrive without booking get a walk-in token at reception
Walk-in tokens are interleaved with pre-booked appointments
Patient can scan a QR code at hospital to check their queue position
3.1.4 Real-Time Queue Tracking
Patient sees: Current Token Being Served | Tokens Ahead | Estimated Wait Time
Automatic SMS notification when patient is 3 tokens away
Patient can wait anywhere in hospital (no need to stand in line)
Live updates every 30 seconds via polling or WebSocket
3.1.5 Appointment History
Patient can view all past and upcoming appointments
Download/share appointment slip as PDF or image
Rate their experience after visit (optional 1–5 star + comment)
3.2 Doctor Module
3.2.1 Daily Schedule View
Doctor sees their token list for the day, sorted by queue order
Each entry shows: Token No. | Patient Name | Age | Chief Complaint (optional)
Doctor can mark consultation as: In Progress | Done | Skip | No Show
3.2.2 Queue Control
Doctor or assistant can: Call Next | Hold | Skip Token | Add Emergency
Estimated per-consultation time is configurable (default: 10 minutes)
Doctor can set status: Available | On Break | Unavailable
3.2.3 Patient Quick Notes
Doctor can add brief consultation note (text only — not a full EMR)
Notes visible to patient in history (optional, doctor configures)
3.3 Receptionist Module
Issue walk-in tokens manually
Check in pre-booked patients (mark as Arrived)
Search patient by phone number or token number
Cancel or reschedule an appointment
View today's complete OPD flow for all doctors
Print token slip (thermal printer support via browser print)
3.4 Hospital Admin Module
3.4.1 Hospital Setup
Add/edit hospital profile: name, address, logo, departments
Create departments (Cardiology, General OPD, Pediatrics, etc.)
Add/edit doctors: name, specialty, availability schedule, avatar
Define time slots per doctor per day (e.g. 09:00–13:00, 20 slots)
Configure holidays, emergency closures
3.4.2 Staff Management
Create receptionist accounts, assign to departments
Role-based access control: Admin, Doctor, Receptionist, View-Only
3.4.3 Reports & Analytics Dashboard
Daily/weekly/monthly OPD count per doctor and department
Average waiting time trend
No-show rate, cancellation rate
Busiest days/hours heatmap
Export reports to Excel/PDF

All data in transit encrypted via TLS 1.3

Queue state persisted in Redis — no queue loss on server restart

Modular codebase — each module (patient, doctor, admin) independently deployable
Centralized logging (ELK Stack or equivalent)
CI/CD pipeline with automated testing (Jest + Cypress)
5. System Architecture
5.1 Architecture Overview
NoWait follows a three-tier web architecture with a decoupled frontend and backend, enabling independent scaling and future mobile app development.
LAYER 1 — PRESENTATION     :  React.js PWA (Patient) + React Admin Dashboard (Hospital)
LAYER 2 — APPLICATION       :  Node.js / Express REST API + WebSocket Server
LAYER 3 — DATA                    :  PostgreSQL (persistent) + Redis (queue/sessions)

Home Screen — Hospital search, Department list, Quick Book CTA
Doctor List Screen — Filter by department, rating, availability
Slot Picker Screen — Calendar + time grid, available slots highlighted
Booking Confirmation Screen — Token card with QR code
Queue Tracker Screen — Live position display, ETA, progress bar
My Appointments Screen — Upcoming and history list
Profile Screen — Personal info, dependents, notifications settings
7.2 Receptionist Screen
Today's Queue Board — Full list of all tokens, sortable, color-coded by status
Walk-In Issue Screen — One-tap token generation for walk-in patients
Patient Search — Look up by phone, name, or token number
7.3 Doctor Screen
Today's List — Compact token queue with patient info
Call Next Token Button — Large, prominent CTA
Patient Notes Popup — Quick text entry per consultation
7.4 Admin Dashboard
Overview Card — Today's OPD count, avg wait, no-shows
Department Management — CRUD for departments and doctors
Schedule Builder — Drag-and-drop or form-based slot configuration
Analytics Page — Charts: daily trend, heatmap, no-show funnel
Staff Accounts — Create/suspend receptionist logins
7.5 Design Principles
Mobile-first: all screens designed for 360px width first, then desktop
Color coding: Green = Done, Orange = In Progress, Blue = Pending, Red = No Show
Loading states: skeleton screens instead of spinners for perceived performance
Error messages: in Nepali with English below; actionable, not technical
8. External Interface Requirements
8.1 SMS Gateway (Sparrow SMS)
Integration: REST API with API key authentication
Message types: OTP, Booking Confirmation, Queue Alert, Reminder, Cancellation
Character limit: 160 chars per SMS; longer messages split automatically
Failure handling: retry 3x with exponential backoff; log failed sends
8.2 WhatsApp Business API (Phase 2)
Provider: Twilio or Meta Business API
Use case: Rich notifications with token card image for feature-phone WhatsApp users
Template messages pre-approved by Meta
8.3 Payment Gateway (Phase 2)
Provider: eSewa or Khalti (Nepal local wallets)
Use case: Optional consultation fee payment at booking
Refund flow for cancellations automated
8.4 HMIS Integration (Phase 3)
REST API connector to existing HMIS (AbhiHMS, MocDoc, etc.)
Bidirectional sync: appointments, patient records (read-only from HMIS)
HL7 FHIR R4 compatibility planned for future
9. Constraints & Assumptions
9.1 Constraints
System does NOT store clinical/medical records (prescriptions, diagnoses) — out of scope for v1
System does NOT manage billing, pharmacy, or laboratory workflows in v1
SMS delivery success depends on network provider — no guaranteed delivery
Initial target: Nepal market only; multi-language beyond Nepali/English is future scope
Browser compatibility: Chrome 90+, Firefox 88+, Safari 14+ (IE not supported)
9.2 Assumptions
Hospital has at least 1 staff member who can manage the admin dashboard
Patients have access to SMS-capable mobile phone (smartphone not required for basics)
Hospitals have a working internet connection (minimum 2 Mbps)
Initial pilot hospitals will provide 1–2 receptionists as system champions during rollout
Auralith Bit will provide onboarding training and 30-day support post-launch
